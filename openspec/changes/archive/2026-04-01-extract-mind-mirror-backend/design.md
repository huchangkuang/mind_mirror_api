## Context

当前后端能力位于同级 `mind_mirror` 项目中，代码组织、鉴权、返回格式、错误处理和数据访问方式存在项目内约定，尚未在 `mind_mirror_api` 中形成统一规范。此次设计目标是在迁移代码的同时，建立稳定的 API 基础设施约束，减少后续模块接入成本，并确保前后端联调行为可预期。

约束条件：
- 技术栈固定为 NestJS + Prisma。
- 输入校验必须使用 `class-validator + class-transformer`。
- 鉴权必须支持 Access Token + Refresh Token 双令牌机制。
- API 路由统一版本前缀 `/api/v1`。
- 所有响应需统一为 `code/message/data`。

主要相关方：后端开发、前端调用方、测试与运维人员。

## Goals / Non-Goals

**Goals:**
- 将 `mind_mirror` 的后端代码安全迁移到 `mind_mirror_api`，形成清晰模块边界。
- 通过全局管道、拦截器、过滤器和守卫统一请求校验、响应格式、异常输出与鉴权行为。
- 建立错误码枚举中心并绑定异常映射，保证业务错误与系统错误输出一致。
- 将数据访问统一收敛到 Prisma，明确 schema 管理、迁移策略与事务使用规范。
- 产出可执行的 API 规范基线，支持后续新增接口按同一标准落地。

**Non-Goals:**
- 不在本次设计中完成全部业务域重构（仅保证迁移后的兼容与标准化）。
- 不引入 GraphQL、gRPC 或多协议网关扩展。
- 不在本次设计中建设全套 API 网关限流与多租户体系。

## Decisions

1. **以 NestJS 分层模块作为迁移目标结构**
   - 决策：按 `module/controller/service/repository`（repository 由 Prisma 封装）重组迁移代码。
   - 理由：降低从旧项目迁移时的耦合，便于后续按领域拆分。
   - 备选：
     - 继续沿用旧目录结构：迁移成本低但技术债延续。
     - 直接做 DDD 全量重构：质量更高但周期与风险不可控。

2. **全局启用 ValidationPipe + DTO 注解校验**
   - 决策：在应用入口统一启用 `transform: true`、`whitelist: true`、`forbidNonWhitelisted: true`，并要求入参 DTO 使用 class-validator 注解。
   - 理由：在控制器前完成数据清洗与校验，减少重复校验代码。
   - 备选：
     - 每个控制器手工校验：灵活但风格不统一且易漏。

3. **鉴权采用 JWT Access + Refresh Token 双令牌**
   - 决策：短时效 access token + 可轮换 refresh token；refresh token 需可撤销（持久化存储 hash 或黑名单）。
   - 理由：兼顾安全性与会话连续性，满足常见登录刷新场景。
   - 备选：
     - 仅 access token：实现简单但用户体验差、风控能力弱。
     - 服务端 session：状态管理复杂，跨服务扩展成本较高。

4. **统一响应与错误处理管线**
   - 决策：成功响应由全局拦截器包装为 `code/message/data`；异常由全局过滤器映射为统一错误码与消息。
   - 理由：前端与测试可依赖稳定结构，避免接口“各自返回”。
   - 备选：
     - 控制器自行返回统一结构：可行但执行一致性弱。

5. **错误码采用集中式枚举并分段管理**
   - 决策：定义 `AppErrorCode`（如 0 成功、1xxxx 参数类、2xxxx 鉴权类、3xxxx 资源类、5xxxx 系统类），异常到错误码映射由统一工厂维护。
   - 理由：提升错误可观测性，便于日志、告警、文档一致对齐。
   - 备选：
     - 仅使用 HTTP status：语义粒度不足，业务可观测性较弱。

6. **API 版本策略固定 `/api/v1`**
   - 决策：Nest 全局前缀使用 `/api`，版本前缀固定 `v1`（可采用 URI versioning）。
   - 理由：与既定规范一致，为后续 `v2` 平滑并行发布留接口空间。
   - 备选：
     - Header 版本：更灵活，但调试与接入可读性较差。

7. **ORM 统一 Prisma 并规范访问边界**
   - 决策：所有数据库访问通过 Prisma Client；复杂事务在 service 层聚合并显式使用事务 API。
   - 理由：简化模型维护与迁移流程，统一 SQL 生成能力。
   - 备选：
     - 混用原生 SQL/多 ORM：迁移初期快，但长期维护复杂度高。

## Risks / Trade-offs

- [迁移过程中行为回归] → 通过接口契约回归测试与关键路径冒烟测试降低风险。
- [统一响应层影响历史调用方解析] → 在迁移窗口期提供兼容层或灰度开关，并同步联调文档。
- [Refresh Token 存储策略不当导致安全风险] → 强制 token 哈希存储、设置过期与轮换、支持主动撤销。
- [错误码枚举扩展失控] → 建立错误码分段规则与评审机制，新增码必须登记。
- [Prisma schema 与历史表结构不一致] → 迁移前先做模型映射清单，分批执行迁移并准备回滚脚本。

## Migration Plan

1. 建立 `mind_mirror_api` 的 NestJS 基础骨架与通用基础设施（响应拦截器、异常过滤器、认证守卫、校验管道）。
2. 从 `mind_mirror` 按业务域分批迁移模块，先迁移认证与用户核心链路。
3. 引入 Prisma schema，并与现有数据库结构对齐；执行首轮迁移与数据验证。
4. 接入 JWT + Refresh Token 全流程（登录/刷新/退出/失效）并通过集成测试。
5. 对迁移接口统一切换到 `/api/v1` + `code/message/data` 响应，完成前端联调。
6. 灰度发布并监控错误码分布、鉴权失败率、接口时延，确认稳定后完成全量切换。

回滚策略：
- 保留旧服务可回退入口；对新增表结构或字段准备反向迁移脚本；通过开关回退到旧鉴权/旧响应适配层。

## Open Questions

- Refresh Token 最终采用数据库持久化还是 Redis 黑名单方案（取决于部署形态与并发规模）。
- 历史接口是否存在必须长期保留的非标准响应格式（如有需制定兼容期限）。
- `mind_mirror` 中是否有不可迁移的运行时依赖（需在正式实施前完成资产盘点）。
